Frequently Asked Questions

1 High Level

1.1 What is the basic architecture?

  • independent database

  • data retrieval via data access layer in interface data-provider<%>, implemented by class db-data-provider%

  • domain entities (e.g., route, stop) modeled as Racket structures (see Data Definitions)

  • GUI implemented via the Racket Graphical Interface Toolkit (https://docs.racket-lang.org/gui)

  • in general, see Architecture Overview

1.2 How is data handled?

1.2.1 In db-data-provider%?

  • data caching in top-level variables

  • routes data can change -> timer-triggered check of table’s checksum; cache update and client callback on change

1.2.2 In the GUI?

  • stop data fully retrieved and cached (no updates; small data volume) -> good performance

  • route data (display between stops or checks before creation) queried via data-provider<%>

  • route creation: data consistency checked as much as possible on gui (button only active if ok); creation via data-provider<%>

2 GUI

2.1 Basic GUI Concept?

  • tabbing for the two big different tasks: display and route creation; see info panel and edit panel

  • bound input whenever possible (sliders for longitude/latitude)

  • hints and deactivated ’Create New Route’ button instead of modal ’you did wrong’ dialog

2.2 Structuring on implementation level?

  • more complicated compound controls glued together from basic Racket controls (e.g., info panel, edit panel, compound stop selector)

3 Racket

3.1 What is Racket?

  • a dialect of the Scheme programming language, which itself is one of the major dialects of the Lisp programming language

3.2 What is special about Racket?

  • dynamically typed

  • fast feedback cycle via REPL (read-evaluate-print loop), i.e. explorative programming without long build/compile cycles

  • extensive library, e.g. for GUI and Web programming, which is very well documented

  • multi-platform (Win/Linux/macOS)
